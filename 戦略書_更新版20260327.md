## 【元の戦略書との主な差異・修正点】

### 誤り1：「Critical Alerts」→「AlarmKit」

元の戦略書は、マナーモード貫通の手段として **Critical Alerts（iOS 12の重大通知機能）** を前提としていた。

**実態**：本アプリは **iOS 26専用の新API「AlarmKit」** を使用している。

| | Critical Alerts（元の想定） | AlarmKit（実際の実装） |
|--|--|--|
| Appleの審査 | 特別なEntitlementが必要・審査が厳しい | 標準機能として利用可能 |
| 通知形態 | Push通知の上位版 | システムアラーム（時計アプリと同等） |
| 音量 | アプリ指定の固定音量 | システムアラーム音量（最大保証） |
| UI | 赤い「！」マーク付き通知 | ロック画面に専用アラームUI |
| 信頼性 | バックグラウンド処理に依存 | OSレベルで保証 |

**→ AlarmKitはCritical Alertsより強力。戦略の方向性（マナーモード貫通）は正しいが、技術的手段が異なる。競合優位の説明もAlarmKitで語り直す必要がある。**

---

### 誤り2：「Apple Intelligence」→「AVSpeechSynthesizer」

元の戦略書は音声読み上げを **Apple Intelligence** によるものとして説明していた。

**実態**：標準iOS API **AVSpeechSynthesizer** を使用している（Apple Intelligenceは不要）。

- ✅ オンデバイス処理（サーバー代ゼロ）→ この点は正しい
- ✅ 完全無料での読み上げ提供 → この点も正しい
- ❌ Apple Intelligence（iOS 18.1+の生成AI機能）は使っていない

**→ 「Apple Intelligenceを活用した」という訴求はAppleガイドライン上グレーゾーン。今後のApp Store申請でも「Apple Intelligenceを活用」とは書かない方が安全。「iOSの音声エンジンによるオンデバイス読み上げ」と表現する。**

---

### 誤り3：アプリ名の理解

元の戦略書は「忘れん坊アラーム」一本での展開を想定していた。

**実態**：3層構造で運用している。

| レイヤー | 名前 | 目的 |
|---------|------|------|
| App Store名 | 忘れん坊アラーム：マナーモード貫通・お薬や予定の忘れ物防止 | ASO（検索流入最大化） |
| ホーム画面表示名 | **ふくろう** | 自己卑下の回避・愛着形成 |
| Siri呼び出しフレーズ | 「**ふくろうにお願い**」「ふくろうで予定を追加」 | 標準アラームとの誤爆防止 |

## 次のフェーズで実装すべき機能（優先順）

### Phase A：v1.1（申請審査通過後・即着手）

**①タスク完了時のポジティブフィードバック** 優先度 🔴

競合「しつこいお薬アラーム」の最大の弱点。完了ボタンを押した瞬間に音声で褒める。

```
「お薬を飲みました！えらいですね。今日もよくできました。」
```

実装方法：アラームをStopした後、短い音声フィードバックをAVSpeechSynthesizerで再生。

---

**②イヤホン切断時の自動ミュート** 優先度 🔴

電車内など公共の場で、イヤホンが抜けた瞬間にアラーム音声を止める。AVAudioSessionのrouteChangeNotificationで実装可能。「音声時計Pro」からの乗り換えを促す差別化点。

---

### Phase B：v1.2（3ヶ月目以降）

**③家族リモートスケジュール設定**

Supabaseはすでに導入済みのため、テーブル設計と権限管理の追加で実装可能。「スマホが苦手な親に代わって子が予定を登録できる」機能は見守り市場で強力な訴求になる。

---

### Phase C：v2.0（6ヶ月目以降）

**④カメラOCRによるお薬手帳・ゴミカレンダー一括読み込み**

Vision frameworkのOCRをオンデバイスで使用。ADHD税（初期設定の手間）を完全撤廃。オンボーディング完了率が劇的に向上する見込み。
