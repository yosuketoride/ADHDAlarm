# 開発ルール（厳守）

## 1. 変更の基本方針
- 指示された課題に必要な最小変更を行うこと。
- 関係のないリファクタリング、UI変更、命名変更は行わないこと。
- まず関連ファイルを読み、呼び出し元・参照先も必要最小限で確認すること。
- ビルドが通る状態を最優先すること。

## 2. デバッグとログ
- 非同期処理、状態更新、API通信の不具合を調査・修正する場合は、必要に応じてデバッグログを追加してよい。
- ログは視認性を高めるため、必ず print("🔄 [クラス名/関数名] 変数状態") のように絵文字（🔄✅❌⚠️）を先頭につけること。
- 問題解決後、不要なログは残しすぎないこと。

## 3. テスト方針
- `SyncEngine`、`OfflineActionQueue`、状態遷移などの裏側ロジックを変更する場合は、可能なら実テストも追加・更新すること。
- テスト追加が難しい場合は、最低限の検証手順を明示すること。

## 4. アーキテクチャ制約
- EventKit を主、AlarmKit を従として扱うこと。
- アプリ起動時・復帰時は `scenePhase == .active` で `SyncEngine.performFullSync()` を呼ぶ前提を壊さないこと。
- `CalendarProviding` / `AlarmScheduling` / `VoiceSynthesizing` / `NLParsing` の各プロトコル経由でアクセスし、Serviceクラスを直接参照しないこと。
- アプリ作成イベントの識別には `Constants.eventMarker(for:)` を使い、マーカーのない EventKit イベントには触れないこと。

## 5. UI・文言
- ユーザー向け文言はやさしい日本語で書くこと。
- IT用語をそのまま出しすぎないこと（例: 同期、デフォルト、プッシュ通知）。
- デザイントークンと既存UIトーンを優先し、勝手に別デザインへ寄せないこと。
- タップターゲットは最低 60pt を意識すること。

## 6. 課金ルール
- コア機能（アラームが鳴ること）を課金壁にしないこと。
- PRO限定機能は `SubscriptionTier` を必ず確認して出し分けること。

## 7. 触ってはいけない領域
- `実装計画書/` 配下の `.md`
- `.github/`
- `CLAUDE.md`
- `AGENTS.md`

## 8. 補足
- 詳細仕様や画面仕様は仕様書の Markdown を必要時に読むこと。
- 技術スタック、ディレクトリ構造、画面仕様は別ドキュメントに分離してよい。

## 9. トリガーコマンド（スキル化）
ユーザーが以下のキーワードを言った場合、指定の行動をとること。

- /plan : 実装前に、アーキテクチャの破綻やiOSの制約による「落とし穴（リスク）」を3つ予想して提示する。
- /ui : ロジックの箱（VStack等）だけを作り、PaddingやColorの細かい装飾はプレビューで人間が調整できるように極力シンプルに出力する。
