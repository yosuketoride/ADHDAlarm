# 状態別アラーム挙動仕様

最終更新: 2026-04-08（現実装に合わせて全体を更新）

## 目的

この仕様書は、忘れ坊アラームの最重要品質である
「アプリがどの状態にあっても、ユーザーが気づける形でアラームが鳴る」
を、実装判断しやすい形で定義するためのものです。

`08_AlarmAudio_実機確認表.md` が実機確認の観点表であるのに対し、
この文書は「各状態で、何を理想挙動とするか」を定義します。

---

## いちばん大事な原則

### 原則1. 最優先は「確実に気づけること」

- アプリ前面かどうかに関係なく、まずはユーザーが気づけることを最優先とする
- 見た目の豪華さや読み上げの自然さより、まず無音にならないことを優先する

### 原則2. 背景・ロック中は OS / AlarmKit を主役にする

- バックグラウンドやロック画面では、アプリ独自UIより OS / AlarmKit による鳴動を信頼する
- その状態で `RingingView` が確実に全面表示されることは前提にしない

### 原則3. 前面時は RingingView を主役にする

- アプリが前面にある場合は、できるだけ `RingingView` を全面表示して強く気づかせる
- 前面時は「音」と「すぐ操作できる画面」をセットで提供する

### 原則4. 音の責務は分けて考える

- `AlarmKit / OS`: 背景・ロック時に確実に気づかせる音
- `RingingView（AVAudioPlayer）`: 前面時のループ再生と読み上げ
- `褒め言葉 TTS`: dismiss 後の AVSpeechSynthesizer による再生（別責務）

### 原則5. 劣化は許容しても、沈黙は許容しない

- 状態によって読み上げやUI演出が弱くなることは許容する
- ただし、無音で気づけない状態は許容しない

### 原則6. .caf が主役。TTS はフォールバック

- AlarmKit に渡す音の第一希望は、事前生成した `.caf`（ビープ＋読み上げ合成）とする
- 前面時も同じ `.caf` を AVAudioPlayer でループ再生する
- `.caf` の生成失敗・ファイル欠落時は TTS（AVSpeechSynthesizer）にフォールバックする
- TTS フォールバックはマナーモード対応のため `.voicePrompt` セッションで再生する
- 読み上げのために確実な鳴動を犠牲にしてはならない

## 現在の音声再生の実装

### 前面時（RingingView が表示されているとき）

1. AlarmKit が発火 → アプリが前面に来た時点で AlarmKit の音は止まる（iOS の仕様）
2. `RingingViewModel.startAudioPlayback()` が起動（4秒ディレイ後）
3. `.caf` ファイルが存在すれば AVAudioPlayer で再生（`.default` セッション・フルボリューム）
4. 再生完了後 5秒後に次のループを開始
5. `.caf` がなければ TTS（`speakAlarmTitle`）にフォールバック（`.voicePrompt` セッション）

### 背景・ロック時

1. AlarmKit が `.caf` をループ再生し続ける（止めるまで継続）
2. ユーザーがアプリを開いた場合は前面時の経路に切り替わる

### .caf の構成（VoiceFileGenerator）

```
ピピピ（880Hz × 3回）→ 2秒無音 → ピピピ × 3回 → 2秒無音 → 読み上げ → 2秒無音
```

- ビープと読み上げを1ファイルに合成することで、背景・ロック時も AlarmKit 経由で両方届く
- 読み上げ部分は生成時にピーク正規化してライブ TTS と同等の音量に揃えている

### dismiss 後の褒め言葉

- `dismiss()` 後に AVSpeechSynthesizer で褒め言葉を再生する（`.default` セッション）
- これは体験向上のための別責務であり、アラーム音とは独立している

## 状態別の実際の挙動

| 状態ID | 状態 | 音を鳴らす担当 | 読み上げ担当 | 備考 |
|---|---|---|---|---|
| ST-1 | アプリ前面のホーム | AVAudioPlayer（.caf ループ） | .caf 内の読み上げ音声 | AlarmKit は前面で止まるため AVAudioPlayer が主役 |
| ST-2 | アプリ前面の設定/入力/別画面 | 同上 | 同上 | RingingView に遷移すれば ST-1 と同じ |
| ST-3 | マイクシート表示中 | AlarmKit → RingingView に遷移 | RingingView 経由 | 入力中でもアラームが勝つ |
| ST-4 | 録音中 | AlarmKit → 録音中断 → RingingView | 可能なら RingingView | 読み上げより可聴性を優先 |
| ST-5 | バックグラウンド | AlarmKit（.caf ループ） | .caf 内の読み上げ音声 | アプリ独自画面より OS 主導でよい |
| ST-6 | ロック画面 | AlarmKit（.caf ループ） | .caf 内の読み上げ音声 | 解除後に RingingView へ接続できるとよい |
| ST-7 | ホーム画面待機 | AlarmKit（.caf ループ） | .caf 内の読み上げ音声 | ST-5 と同様 |
| ST-8 | スワイプキル後 | AlarmKit（.caf ループ） | .caf 内の読み上げ音声のみ期待できる | 前面復帰（ケース1）は起きない |
| ST-9 | 再起動直後 | AlarmKit または復旧処理 | 状況次第 | 予定保持と再接続が重要 |
| ST-10 | 電源OFF後 | 復帰後処理 | 復帰後処理 | 発火保証は前提にせず、復帰後に埋もれないことを重視 |
| ST-11 | 通知を閉じた後 | 状態に応じて次の導線 | 状態に応じる | `awaitingResponse` 相当の扱い |
| ST-12 | スヌーズ後 | AlarmKit（再登録） | .caf 内の読み上げ音声 | 通常アラームと同品質で再鳴動 |

## .caf と TTS フォールバックの判定

```
voiceFileName が存在する
  └→ ファイルが存在 かつ サイズ > 0
        └→ YES: AVAudioPlayer で .caf 再生（.default セッション）
        └→ NO:  speakAlarmTitle()（TTS / .voicePrompt セッション）
  └→ NO:  speakAlarmTitle()（TTS / .voicePrompt セッション）
```

- `.voicePrompt` はマナーモード中に音が出るために必要（実機確認済み）
- `.caf` 再生は `.default` セッション（フルボリューム）。`.voicePrompt` にすると iOS が音量を自動で下げるため使わない

## 仕様と技術制約の切り分け

### 仕様として守ること

- 前面・背景・ロック中のいずれでも、ユーザーが気づける鳴動を目指す
- 前面時は、できるだけ `RingingView` に遷移する
- 通知を閉じた後も、未対応状態がホーム上で埋もれないようにする
- スヌーズ後は、次回時刻で同等品質の再鳴動を目指す

### 技術制約として受け入れること

- AlarmKit はアプリが前面に来た時点で音を止める（iOS の仕様）。前面時は必ず AVAudioPlayer が音の主体になる
- ロック画面やバックグラウンドでは、アプリ独自の全面UIや複雑な音制御は制限を受ける
- `AlarmKit` は実機確認が前提で、シミュレータでは信頼できない
- 音声ファイル再生、TTS、録音中AudioSession は状態により競合しうる
- 端末電源OFF中は、発火そのものより「復帰後どう扱うか」が重要になる
- AVAudioPlayer と AlarmKit は使うボリュームチャンネルが異なる（前者はメディア、後者はアラーム）が、`.default` セッションでフルボリューム再生すれば両者の音量は揃う

## 画面遷移の設計方針

### 前面時

- 発火時に可能なら `RingingView` を全面表示する
- 現在の画面がホーム、設定、入力中のどれであっても、原則として鳴動を優先する

### 背景時

- まずは OS / AlarmKit の通知導線で気づけることを優先する
- アプリに戻ってきた時に、必要なら `RingingView` や未対応状態へ接続する
- アプリ復帰時に毎回即 `RingingView` を再開して再読み上げしない
- 次の条件を満たす時だけ `RingingView` の再開候補とする
  - 本当に鳴動継続中である
  - 予定時刻から大きく外れていない
  - 直前に OS 経由で通知済みではない、または再開が必要である

### ロック画面時

- ロック解除後に、同じアラームが二重に再開しないよう制御する
- 一方で、未対応のまま消えてしまうことも避ける

## 状態遷移との関係

### 状態の考え方

- `scheduled`: 未発火・待機中
- `alerting`: 通知中
- `awaitingResponse`: 通知後・未対応
- `completed`: とめる
- `skipped`: 今日は休む
- `snoozed`: スヌーズ後の再待機
- `missed`: 未対応確定、または別途定義された未達状態

### `awaitingResponse` のトリガー

- `RingingView` が `dismiss / skip / snooze` を経ずに閉じられた場合に `awaitingResponse` とする
- これは「通知は行われたが、ユーザーの明示操作がまだ無い」状態を表す
- iOS 側のすべての閉じ方を完全に検知できるとは限らないため、実装では「確実に観測できる閉じ方」から優先して扱う

## この仕様から見た実装上の判断基準

### 判断基準1. まず「どの状態で」起きる問題かを切り分ける

- 前面ホームだけ壊れるのか
- ロック画面だけ壊れるのか
- 録音中だけ壊れるのか

を分けて考える。

### 判断基準2. UIの豪華さより、可聴性と復帰導線を優先する

- 前面以外で `RingingView` の再現度を無理に上げるより
  OS導線 + 復帰後の状態復元を優先する

### 判断基準3. 「誰が音を鳴らすか」を曖昧にしない

- `AlarmKit / OS` が鳴らす音（背景・ロック時）
- `AVAudioPlayer` が鳴らす音（前面時の .caf ループ）
- `AVSpeechSynthesizer` が鳴らす音（TTS フォールバック・褒め言葉）

の責務を明確に分ける。

## 現時点での設計メモ

- 本アプリでは、背景・ロック時の確実な鳴動が最優先である
- そのうえで、前面時には `RingingView` による強い通知体験を目指す
- もし前面時の体験と背景時の体験が衝突する場合は、まず背景時の確実性を守る
- ただし、前面時に本来できるはずの `RingingView` / 気づく音 / 読み上げ が弱くなっている場合は、回帰として扱ってよい

## 既知の逸脱

### KD-2. ロック画面や復帰直後に、同一アラームの二重接続が起こる可能性がある

- `08_AlarmAudio_実機確認表.md` でも、ロック画面後に再度同じアラームが鳴る事象が記録されている
- 状態復元時は「埋もれないこと」と「二重再開しないこと」を両立する必要がある
